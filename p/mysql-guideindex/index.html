MySQL 索引

索引优化

2025-11-25

深度硬核：MySQL 索引全家桶，从底层 B+ 树到千万级调优

在后端开发中，MySQL 索引是面试的“重灾区”，也是性能调优的“杀手锏”。很多人知道索引能让查询变快，但不知道它为什么快，更不知道如何避坑。本文将带你深度复盘 MySQL 索引的每一个细节，从底层数据结构到千万级场景调优。

一、 为什么选 B+ 树？

MySQL 的默认存储引擎 InnoDB 选择 B+ 树作为索引结构，并非偶然，而是对磁盘 IO 特性权衡后的最优解。

1. B+ 树 vs B 树

特性 B 树 B+ 树
节点存储内容 非叶子节点、叶子节点均存储整行数据 非叶子节点仅存索引键（Key），叶子节点存完整数据
空间利用率 低，相同页面能存储的索引指针少 高，页面容纳更多指针，树的“出度”更大
范围查询效率 需中序遍历，效率低 叶子节点组成双向有序链表，范围查询直接“平推”

核心差异：B+ 树把数据都放到叶子节点，非叶子节点只做索引指引，这让树的高度大幅降低。

2. 磁盘 IO 的计算

磁盘 IO 是数据库查询的核心性能瓶颈，而 B+ 树的高度直接决定 IO 次数。

假设一个高度为 3 的 B+ 树：

  1. 根节点常驻内存，无需 IO；
  2. 第一次 IO：读取二级索引节点，定位目标叶子节点的位置；
  3. 第二次 IO：读取叶子节点，获取完整数据。

结论：千万级数据量下，B+ 树通常只需要 2-3 次磁盘 IO 即可获取数据，而全表扫描可能需要百万次 IO，性能差距达数个数量级。

二、 索引的分类体系

从不同维度拆解索引，能更清晰地理解其设计逻辑和使用场景。

1. 物理存储维度

这是索引的底层分类，直接决定数据的存储和查询方式。

聚簇索引（Clustered Index）

  • 本质：数据即索引，索引即数据。行数据直接存储在索引的叶子节点上。
  • 特性：每张 InnoDB 表必须且只能有一个聚簇索引。
  • 选择规则：
    1. 优先使用主键作为聚簇索引；
    2. 无主键时，选择唯一非空索引；
    3. 无上述索引时，InnoDB 自动生成隐藏的 row_id 作为聚簇索引。

二级索引（Secondary Index / 非聚簇索引）

  • 本质：叶子节点不存储整行数据，只存储聚簇索引的键值（通常是主键值）。
  • 回表（Look-up）：通过二级索引查到主键后，需要再去聚簇索引中查询完整数据，这个过程就是回表。
  • 性能影响：回表会增加一次磁盘 IO，是二级索引查询的性能损耗点。

2. 字段特性维度

从索引字段的数量和约束条件划分，对应不同的业务场景。

索引类型 核心特性 适用场景
主键索引 唯一、非空，默认作为聚簇索引 表的唯一标识，如用户 ID、订单 ID
唯一索引 值不能重复，允许 NULL（多个 NULL 不冲突） 唯一标识字段，如手机号、身份证号
联合索引 多个字段组合而成的索引 多字段组合查询，如 name + age、user_id + create_time
普通索引 无唯一性约束，最基础的索引类型 普通查询字段，如文章标题、商品分类

三、 灵魂法则：最左匹配原则

联合索引的核心使用逻辑，也是索引调优的“第一道门槛”。

假设我们创建了联合索引 INDEX(a, b, c)，可以把它想象成一本按“姓→名→年龄”排序的电话簿：查人时必须先按姓氏找，姓氏确定后再找名字，最后找年龄。跳过前面的字段，直接查后面的字段，索引就会失效。

1. 匹配规则实战

查询条件 索引匹配情况 原理分析
WHERE a=1 AND b=2 AND c=3 全匹配（a,b,c 均生效） 完全符合索引的字段顺序
WHERE a=1 AND b=2 部分匹配（a,b 生效） 匹配最左前缀，c 未使用但不影响前面字段
WHERE a=1 AND c=3 部分匹配（仅 a 生效） b 字段断层，c 无法使用索引
WHERE a=1 AND b>2 AND c=3 部分匹配（a,b 生效，c 失效） 范围阻断：b 是范围查询，后面的字段索引失效
WHERE b=2 AND c=3 完全失效 缺少最左前缀 a，无法触发索引

2. 关键补充

  • 优化器自动排序：WHERE b=2 AND a=1 会被优化器调整为 a=1 AND b=2，不影响索引生效。
  • 覆盖索引例外：如果查询字段仅包含索引字段（如 SELECT a,c FROM t WHERE b=2），且索引为 (a,b,c)，可能触发覆盖索引，无需遵循最左匹配。

四、 索引失效的“避坑指南”

很多时候索引“形同虚设”，都是因为踩了这些常见陷阱：

  1. 对索引列做运算或函数操作

    1
2
3
4

    -- 错误：age 字段做了加法运算，索引失效
WHERE age + 1 = 10
-- 错误：使用 SUBSTRING 函数，索引失效
WHERE SUBSTRING(name, 1, 3) = 'Tom'

    优化方案：运算逻辑放到等号右边，或避免在索引列上用函数。

  2. 隐式类型转换

    1
2

    -- 错误：mobile 是 varchar 类型，传入数值会触发隐式转换
WHERE mobile = 13800138000

    优化方案：字符串字段查询必须加引号，写成 WHERE mobile = '13800138000'。

  3. 左模糊查询

    1
2
3
4

    -- 错误：% 放在左边，索引失效
WHERE name LIKE '%Tom'
-- 正确：% 放在右边，索引生效
WHERE name LIKE 'Tom%'

  4. OR 条件的“坑” OR 两边的字段必须都有索引，否则整个查询会走全表扫描。

    1
2

    -- 假设只有 a 有索引，b 无索引，索引失效
WHERE a=1 OR b=2

  5. 不等于和 IS NOT NULL

    • !=/<> 对数值类型字段查询时，大概率失效；
    • IS NOT NULL 无法使用索引，而 IS NULL 可以。

五、 高阶优化：实现千万级数据“丝滑”查询

1. 覆盖索引：告别回表的“终极优化”

定义：如果查询的字段全部包含在索引中，MySQL 无需回表，直接从索引中返回数据，这就是覆盖索引。 判断标志：EXPLAIN 结果的 Extra 列显示 Using index。

案例

1
2
3
4
5
6
7

-- 需求：查询用户姓名和年龄，条件是姓名=张三
-- 原始表结构：id(主键), name, age, address

-- 方案1：无覆盖索引，需要回表
SELECT name, age FROM user WHERE name='张三';
-- 方案2：创建联合索引 INDEX(name, age)，触发覆盖索引
SELECT name, age FROM user WHERE name='张三';

性能提升：减少一次回表 IO，查询效率提升 10 倍以上。

2. 索引下推：减少回表次数的“利器”

MySQL 5.6 及以上版本支持索引下推（ICP），核心逻辑是：在遍历二级索引时，提前过滤不符合条件的记录，减少回表次数。

案例 索引为 INDEX(name, age)，查询 WHERE name='张三' AND age > 20：

  • 无索引下推：先通过 name 找到所有“张三”的主键，回表后再过滤 age > 20；
  • 有索引下推：遍历索引时，直接过滤 age > 20 的记录，再回表查询，减少回表次数。

3. 用 EXPLAIN 分析索引使用情况

EXPLAIN 是分析索引的“神兵利器”，重点关注以下 4 列：

列名 核心作用 优化目标
type 索引匹配类型 至少达到 range 级别，最优是 const/eq_ref
key 实际使用的索引 与预期索引一致，否则说明索引失效
rows 预估扫描行数 数值越小越好，接近实际结果最佳
Extra 额外信息 出现 Using index 是最优；出现 Using filesort/Using temporary 需优化

type 优先级排序： system > const > eq_ref > ref > range > index > ALL

六、 总结与建议

1. 索引口诀（必背） 

1
2
3
4

带头大哥不能死，中间兄弟不能断；
索引列上不计算，like 百分加右边；
范围后面全失效，字符串查询带引号；
OR 两边都有索引，覆盖索引显神威。

2. 一般性建议

  • 索引不是越多越好：新增索引会降低插入、更新、删除的性能，需在查询效率和写入效率之间平衡。
  • 联合索引字段顺序：过滤性越强的字段越靠前（如性别字段过滤性弱，应放在后面）。
  • 优先使用覆盖索引：减少回表操作，是提升查询性能的最优解之一。

3. SQL 优化步骤

  1. 监控慢查询：开启慢查询日志，抓取超过阈值的 SQL（如执行时间 > 1s）；
  2. EXPLAIN 分析：通过 EXPLAIN 判断索引是否生效、是否有文件排序；
  3. Profile 深入排查：用 SHOW PROFILE 查看 SQL 执行的详细生命周期；
  4. 参数调优：结合业务场景调整 MySQL 配置（如 innodb_buffer_pool_size）。
